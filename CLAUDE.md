# 專案開發規範

## 文件結構
- SRS: /docs/SRS.md（需求規格）
- SDD: /docs/SDD.md（設計文件）
- Workflow: /docs/workflows/（流程圖與業務流程）
- 程式碼: /src/

## 新增或修改需求的工作流程（強制遵守）

當使用者提出任何新需求、變更需求、或修改現有功能時，你必須依照以下順序執行，**不可跳過任何步驟**：

### Step 1: 更新 SRS
- 先閱讀 `/docs/SRS.md` 了解現有需求
- 在對應章節新增或修改需求項目
- 標註需求編號（例如 FR-015）、變更日期、變更原因
- 完成後向使用者確認 SRS 修改內容是否正確

### Step 2: 更新 SDD
- 在使用者確認 SRS 後，才能進行 SDD 修改
- 閱讀 `/docs/SDD.md`，找出受影響的模組、API、資料庫設計
- 更新對應的設計章節，並標註對應的 SRS 需求編號
- 如涉及流程變更，同步更新 `/docs/workflows/` 中的流程圖
- 完成後向使用者確認 SDD 修改內容

### Step 3: 修改程式碼
- 僅在 SRS 與 SDD 都更新完成且使用者確認後，才可修改程式
- 程式註解中需標註對應的 SRS / SDD 章節編號
- 撰寫或更新對應測試

### Step 4: 更新 CHANGELOG
- 在 `/CHANGELOG.md` 記錄本次變更，包含 SRS / SDD / Code 三個層級的變動摘要

## 禁止事項
- ❌ 不可在未更新 SRS 的情況下直接修改 SDD 或程式
- ❌ 不可在未更新 SDD 的情況下直接修改程式
- ❌ 若使用者要求跳過某步驟，請先提醒此規範，並請使用者明確確認跳過

## 例外情況
僅以下情況可跳過文件更新：
- 修正錯字、格式調整
- 修補不影響功能的 bug（但仍需在 CHANGELOG 註記）
- 重構（不改變外部行為）
